# 开发文档规范 V2.4

> 目的：统一开发文档的目录结构、文件命名、体积上限、必含小节。让每次新 AI 会话无需猜测文档组织方式。
> 本文件是固定加载项（~3KB），通过定义规范来**减少后续按需加载的文档体积**。

---

## 一、目录结构约定

```
开发文档/
├── 00-必读/           — 规范、契约、自检清单、AI 上下文速览
│                        每个 AI 会话启动时优先加载此目录
├── 01-变更日志/       — 每日变更日志 CHANGELOG-YYYY-MM-DD.md + 汇总索引
├── 02-项目参考/       — 工具添加说明、账号密码一览、项目结构说明
├── 03-架构设计/       — 架构分析报告、设计系统、组件 API、状态管理
├── 04-方案设计/       — 按主题分子目录：Agent/、导师/、技术/、运营/、待执行/
├── 05-产品与合规/     — PRD、合规文档、备案材料、商业评估
├── 06-软著与审核/     — 软著源码 PDF、审核提交材料
├── 07-Agent内容模板/  — Agent 提示词模板（按 P0 核心角色 / 通用角色分类）
├── 08-接口与交互/     — API 文档、交互规范、API 速查表
├── 归档/              — 不再使用的文档按季度归档（如 2026-Q3/）
└── 开发者自创工具/    — Vibe Coding Rules、导师生成器等工具
```

### 编号含义速记

| 编号 | 内容 | AI 何时加载 |
|:--|:--|:--|
| 00 | 规范/契约/自检 | 每次会话必读 |
| 01 | 变更日志 | 按日期按需 |
| 02 | 项目参考 | 首次接触项目 / 新增工具 |
| 03 | 架构设计 | 做大改动前 |
| 04 | 方案设计 | 做方案决策前 |
| 05 | 产品与合规 | 产品规划 / 备案 |
| 06 | 软著与审核 | 申报软著时 |
| 07 | Agent 模板 | 修改 Agent 提示词 |
| 08 | 接口与交互 | 改路由 / 改前端交互 |

## 二、文件规范

### 命名规则

- **使用中文描述性名称**，让人类一眼看懂文件内容
- 方案文件前缀 `待执行-` 表示方案已设计但未实施（如 `待执行-2026-07-05-微信支付安全加固.md`）
- 变更日志固定格式：`变更日志-YYYY-MM-DD.md`
- 归档文件前缀 `YYYY-QN/` 按季度归档

### 体积上限

```
硬上限：单文件 ≤ 500 行（约 20KB）
  - 代码红线 300 行，文档放宽至 500 行
  - 超过上限 → 必须按子主题拆分为多个文件
  - 原文件保留为索引（列出子文件清单+每个子文件的摘要）

豁免：
  - PDF/图片/视频附件不计入（不是 markdown，不会被加载到 AI 上下文）
  - 代码示例超过 50 行不算在 500 行内（但也不会被 AI 完整阅读）
```

### 拆分规则

当文件超过 500 行时：
1. 按子主题拆分为独立文件
2. 原文件改为索引页，格式：`主题概述 + 子文件清单（文件名 + 一句话摘要）`
3. 子文件命名：`{原文件名}-{子主题}.md`
4. 示例：
   ```
   分层执行引擎.md  ← 索引（200行）
   分层执行引擎-架构设计.md  ← 子文件（400行）
   分层执行引擎-实现细节.md  ← 子文件（350行）
   分层执行引擎-测试方案.md  ← 子文件（300行）
   ```

### 编码

- 所有文档统一 UTF-8 编码
- 行尾使用 LF（`\n`）

## 三、必含小节规范

不同类型的文档必须包含以下小节。缺失任一小节 → 自检阻断。

### 所有文档通用（V2.4.1 新增）

每个开发文档的**表头**必须包含管理规范引用：

```markdown
# {文档标题}

> 📋 管理规范见: 开发文档/00-必读/开发文档管理.md
```

`📋` 行让 AI 在打开任何文档时知道去哪里找文档创建/更新规则。
缺失此引用 → 自检 ⚠️ 警告（规则26.g）。

### 方案设计文档（04-方案设计/）

```markdown
## 背景/问题
（为什么需要这个方案）

## 方案对比
（列出 ≥2 个备选方案，每个方案注明优缺点）

## 选择理由
（@WHY 选了方案X而不是方案Y：具体原因）

## 实施计划
（分步骤，标注预估时间）

## 实现位置 ← V2.3 强制要求
- 功能点A: `server/routes/xxx.js#L10-L50`
- 功能点B: `public/js/xxx.js#L100-L200`
```

### 架构设计文档（03-架构设计/）

```markdown
## 现状分析
（当前架构的痛点和瓶颈）

## 设计方案
（新架构的描述、图示、关键模块）

## 关键决策
（每个重大决策用 @WHY 格式标注理由）

## 实现位置
（所有相关代码文件的清单）
```

### 接口文档（08-接口与交互/）

```markdown
## 路由
（HTTP 方法 + 路径 + 描述）

## 请求参数
（参数名 | 类型 | 必填 | 说明）

## 响应格式
（成功/失败的 JSON 结构）

## 错误码
（错误码 | 含义 | 处理建议）

## 调用方
（哪些前端组件/文件使用了此接口）
```

### 变更日志（01-变更日志/）

```markdown
## 改动概述
（一句话说明本次做了什么）

## 改动文件清单
（文件路径 | 改动类型(新增/修改/删除) | 简要说明）

## 新增依赖
（如有新 npm 包或其他依赖）

## 自检结果摘要
（通过项数 / 发现及修复的问题数）
```

## 四、文档与代码联动规则（V2.3+ / V2.4.1 四级链式闭环）

### 代码 → 文档（单向索引）

代码中通过 `📖` 标签指向开发文档：

```js
// 📖 完整设计: 开发文档/03-方案设计/支付回调重试方案.md
```

### 文档 → 管理文档（V2.4.1 新增）

每个文档表头通过 `📋` 标签指向管理文档：

```markdown
# 支付回调重试方案

> 📋 管理规范见: 开发文档/00-必读/开发文档管理.md
```

这是链式驱动的第二跳——AI 从代码跳到文档后，通过 `📋` 跳到管理文档，获得全局规则。

### 文档 → 代码（反向索引）

被引用的文档必须在末尾包含「实现位置」小节，列出所有相关代码路径：

```markdown
## 实现位置
- 幂等键生成: `server/routes/api/payment.js#L142-L158`
- 重试队列: `server/scheduler/retry-queue.js#L30-L85`
- 前端展示: `public/js/chat-core.js#L1021-L1050`
```

### 管理文档 → 决策树（V2.4.1 新增）

管理文档（`00-必读/开发文档管理.md`）的第三节「改动→文档决策树」是链式驱动的第三跳——AI 获得全局规则后，通过决策树确定具体要改哪些文档、是创建还是追加还是拆分。

### 联动的自检规则

| 检查项 | 不通过后果 |
|:--|:--|
| 📖 路径指向的文档存在 | 不存在 → 阻断（规则 26.a） |
| 被引用文档有「实现位置」小节 | 缺失 → 阻断（规则 26.b） |
| 被引用文档有 📋 管理规范引用 | 缺失 → ⚠️ 警告（规则 26.g） |
| 管理文档 `开发文档管理.md` 存在 | 不存在 → 阻断（规则 26.h） |
| 代码改动 + 📖 引用 → 文档是否需更新 | 陈旧 → ⚠️ 警告 |
| 「实现位置」中的代码路径存在 | 不存在 → ⚠️ 警告 |
| 文档改动 → 管理文档索引是否更新 | 未更新 → ⚠️ 警告（规则 26.i） |

### 完整链式路径（四跳闭环）

```
代码 @COUPLED / 📖           ← 第1跳：AI 打开代码，看到锚点
    ↓
文档 📋 管理规范见: ...       ← 第2跳：AI 打开文档，看到 📋
    ↓
管理文档 决策树              ← 第3跳：AI 获得全局规则 + 影响面
    ↓
管理文档 文档索引             ← 第4跳：AI 改完文档后更新索引
    ↓
自检验证 四环闭合             ← 闭环：自检扫描所有链是否断裂
```

AI 每一步只加载 3-5KB 上下文，但通过链条形成完整信息覆盖。

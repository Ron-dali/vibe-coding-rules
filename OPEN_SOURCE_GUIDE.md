# Vibe Coding Rules 开源手册

> 从聊天记录提炼的项目说明、独特价值、设计哲学、和为何值得关注。
> 面向 GitHub 开源场景，让路过的人 30 秒内看懂"这是什么"和"为什么我需要它"。

---

## 一句话定义

**这不是一个 Skill，这是一套给 AI 编程助手的 QA 流水线。**

它不是教 AI 怎么处理 PDF、怎么生成 Excel——那是"领域技能"。它做的是另一件事：**确保 AI 每次改代码后，项目不会烂掉。**

---

## 核心答案：为什么你需要它？

### 问题：AI 编程助手的"失忆症"

你用 AI（Cursor / CodeBuddy / Claude Code / Windsurf）写代码时，最大的痛点是什么？

1. **AI 修了 A 处，炸了 B 处** —— 它不知道这两个文件是联动的
2. **同一个 Bug 反复出现** —— 因为人类没来得及把上次的教训写进规则
3. **项目越写越乱** —— AI 每次会话从零开始，不知道项目有什么坑
4. **文档和代码脱节** —— AI 生成了漂亮的文档，但代码改了以后文档万年不更新

### 我们的解法：一套"自动驾驶"的代码质量保障系统

```
改代码前   →  五大原则 + 🍞面包屑预扫描（AI 先看地图再动手）
改代码中   →  终端命令安全封装（防死循环、超时、恶意命令）
改代码后   →  28条自检规则（按改动类型自动匹配）+ 自动种面包屑
              →  浏览器自动化测试（DOM + 截图 + OCR）
              →  自动写变更日志
              →  规则自动生长（新Bug → 新规则）
              →  git commit → 闭环 ✅
```

**一句话**：你的 AI 每改一次代码，这个系统自动检查一次。它会把踩过的坑记住，下次不再踩。它会在每个文件种下"面包屑"，让后来的 AI 打开文件就知道改了这里会影响谁。

## A. 这个工具是什么？

**一个 AI 编程助手的 QA（质量保障）元框架。**

- **不是**：PDF 处理、Excel 生成、PPT 创建等"领域技能"
- **不是**：Cursor Rules 集合（告诉你"用 React"、"用 TypeScript"）
- **是**：编程流程本身的质量控制系统——无论你用任何 AI 工具、写任何语言

### 和其他 GitHub 高星项目的区别

| 项目 | 定位 | Stars | 和我们关系 |
|------|------|-------|-----------|
| **anthropics/skills** | 官方 Skill 仓库 | 138k | 他们是"语言标准"，我们是"最佳实践" |
| **claude-skills** | 354个领域技能市场 | 5,200+ | 互补——他们的技能装进我们的流水线 |
| **awesome-cursorrules** | Cursor 规则集合 | 40k+ | 他们管"怎么写"，我们管"写了别炸" |
| **Vibe Coding Rules** | AI编程QA元框架 | N/A | **市场上唯一做这件事的** |

### 唯一性：为什么没有竞品？

因为所有 Skill 生态都在做"横向扩展"——更多领域、更多工具。我们在做"纵向深化"——让 AI 的编程过程本身受管控。

**这个方向天然缺位**：人类程序员有 review、CI/CD、lint、test、changelog，但 AI 程序员没有——AI 改完代码直接 commit，中间没有任何质量闸门。我们在填补这个空白。

## B. 核心能力拆解

### 1. 🍞 面包屑自动播种（V2.5 杀手特性）

每次改代码，自动在文件头部种下面包屑：

```
/*
 * ─── 🍞 AI Breadcrumb Navigation ──────────────────
 * Tag meanings: @COUPLED=linked files @GOTCHA=gotcha @BUGFIX=bug fix @MAGIC=magic number
 *
 * Breadcrumbs (changing this affects):
 *   @COUPLED sync needed: fileA.js, fileB.html
 *   @GOTCHA don't parallelize: server rate-limits to 5 concurrent
 *   📖 see: docs/payment-design.md
 * ──────────────────────────────────────────────────
 */
```

**为什么这是杀手特性**：
- 面包屑从"人工刻意耕种"变成"每次改代码自动播种"
- 🍞 头部内嵌标签含义表——任何 AI 即使没加载本 Skill 也能理解
- 面包屑覆盖率 = 代码改动覆盖率（改到哪就种到哪，不需全量标注）
- 从零开始 → 项目写完即全覆盖；半路接手 → 热点代码优先覆盖

### 2. 信任分级规则自生长（V2.1 开始）

你的项目会"记住自己踩过的坑"：

| 级别 | 名称 | 行为 | 晋升条件 |
|------|------|------|----------|
| Tier 0 | 观察池 | 仅记录 | 首次发现 |
| Tier 1 | [SOFT] 软规则 | 警告可放过 | 置信度 ≥ 3 |
| Tier 2 | [HARD] 硬规则 | 阻断必改 | 置信度 ≥ 5 |

误报过多自动退役，无需人工审核。全程记录在 `growth-log.md`。

**核心价值**：Bug 修复不是终点，是规则的起点。同一个 Bug 永远不会出现第二次。

### 3. 四环链式闭环（V2.4.1）

```
代码注释 (@COUPLED / 📖)
    ↕ 第1跳：AI 打开代码，看到锚点
文档 (「实现位置」小节)
    ↕ 第2跳：AI 打开文档，看到 📋
管理文档 (决策树 + 索引)
    ↕ 第3跳：AI 获得全局规则 + 影响面
自检系统 (规则25/26/27/28)
    ↕ 第4跳：验证四环闭合
```

**核心洞察**：AI 不需要了解整个项目，只需要顺着面包屑找到相关的那几个文件。每一步只加载 3-5KB 上下文。

### 4. 28条代码纪律规则

来自真实项目的血泪教训。按改动类型自动匹配，不改不用查：

| 改了什么 | 触发哪些规则 |
|----------|-------------|
| 新增工具 | Schema完整性 + 截断 + 面包屑 |
| 前后端JS | `||`吞0 + select假选中 + 意图锚点 |
| Express路由 | 固定路由在前 + GET/POST对称 + 文档联动 |
| 弹窗/浮层 | 穷举所有关闭路径 |
| 数据库 | 迁移编号 + 面包屑 |

完整清单见 `self-check/references/self-check-full.md`

### 5. 设计意图锚点（V2.3）

10 种标准化注释标签，让代码"自带说明书"：

`@WHY` `@GOTCHA` `@MAGIC` `@COUPLED` `@BUGFIX` `@PERF` `@ASSUME` `@TODO` `@EXPIRE` `📖`

后来的 AI 打开代码文件，无需额外喂文档，一眼就知道"为什么这么写"。

## C. 设计哲学（为什么这么设计）

### 哲学 1：AI 不需要了解整个项目，只需要顺着面包屑找到相关文件

这是整个 V2.4.1 四环闭环的核心理念。我们不追求让 AI 一次性加载全量文档（那会超出上下文窗口），而是让 AI 像人类程序员一样：**看代码注释 → 顺着链接找文档 → 文档里又有链接找更多**。每一步只加载 3-5KB，永远不溢出。

### 哲学 2：面包屑从"刻意耕种"变成"自动播种"

V2.5 之前，面包屑需要人工刻意维护——就像写 Wiki 需要人工更新一样，最终都会腐烂。V2.5 的核心突破：**改代码的过程就是种面包屑的过程**。不需要额外投入，覆盖率自然增长。

### 哲学 3：自解释文档——AI 不加载 Skill 也能理解

🍞 头部内嵌了标签含义速查表。即使 AI 没有加载我们的 coding-principles 或 self-check Skill，只要看到 🍞 块，就能立刻理解 `@COUPLED` 指联动文件、`@BUGFIX` 是 Bug 修复记录。

### 哲学 4：规则不是写出来的，是长出来的

传统的 lint 规则是人写的，只有人能想到的场景。但你的项目有独特的坑——微信支付的超时、特定 API 的并发限制、某个字段的特殊格式。这些坑只有踩过才知道。V2.1 的信任分级系统让这些坑自动生长为规则。

### 哲学 5：从 TBP 项目脱壳——保证通用性

这个工具不是凭空设计的。它来自一个真实项目（TBP 躺不平联盟）3 个月的实战。V1 是 TBP 专用版，V2 完全脱壳为任何项目可用。每次新增规则都会问："这条规则是 TBP 特有的，还是所有项目都会遇到的？"只有后者才进入通用版。

V2.6 与 TBP 专属版在三层纪律体系（前置门→六原则→三陷阱）上完全对齐，差异仅在：
- TBP 专属版的附录案例引用 TBP 项目自身的文件路径和历史 Bug
- 通用版的附录案例使用泛化的通用场景描述
- 通用版不含 TBP 专属的"300行红线豁免"中的项目特定文件

## D. 这个项目的起源故事（讲故事的力量）

2026 年初，一个非程序员在 AI 辅助下开发了一个叫做 TBP（躺不平联盟）的全栈项目——微信支付、多用户系统、实时聊天、工作流引擎都涉及。

AI 写代码很快，但有几个问题：

1. **改 A 炸 B**：修了聊天页的 Bug，管理后台的统计图表挂了——因为两个页面共享同一段数据逻辑
2. **同一个 Bug 修了三次**：微信支付回调超时导致重复扣款，每次都是"这次一定不会再出现了"，但下个版本又回来了
3. **AI 每次会话都是"新人"**：换了新会话，AI 完全不知道之前为什么要那样写，把之前刻意避开的坑又踩一遍
4. **文档成了僵尸**：每次 AI 改完代码都说"我会更新文档"，但永远"回头再补"

于是开始建立规则——每发现一个 Bug 就写一条检查规则。规则越积越多，发现可以用 AI（对，用 AI 管 AI）来自动执行这些检查。

三个月后，这套规则已经进化了 5 个大版本：
- V2.1：规则自动生长（Bug → 规则，不再手动维护）
- V2.3：设计意图锚点（10 种标签让代码自带说明书）
- V2.4：注释-文档-自检闭环（改了代码自动提醒更新文档）
- V2.4.1：四环链式闭环（AI 顺着面包屑一步步导航）
- V2.5：🍞 面包屑自动播种（不需要人工维护，改代码即种地图）

**V2.5 的核心理念**："面包屑从刻意耕种变成自动播种"——改代码的过程，就是让 AI 在项目里建地图的过程。

## E. 安装和使用

```bash
# 一键安装（Windows）
install.bat

# macOS / Linux
chmod +x install.sh && ./install.sh

# PowerShell
.\install.ps1
```

零依赖即可使用核心功能。Playwright + tesseract.js 是可选依赖（仅 web-testing 需要）。

安装后，Agent Memory 自动写入触发器。下次说"我要做一个XX"时，Agent 会询问是否初始化流水线。

## G. 开源价值主张

### 对个人开发者

你的 AI 编程助手不再"失忆"。每次改代码都有质量闸门，频繁出现的 Bug 会自动变成规则。

### 对团队

统一 AI 编程规范。新成员（人或 AI）接手项目时，🍞 面包屑让它们秒懂项目结构。

### 对 Skill 生态

这不是第 355 个"PDF处理"Skill。这是一个全新的品类——**AI 编程 QA 元框架**。任何领域 Skill 都可以在这套流水线上运行。

## H. 面向开源社区的说明

如果你是路过这个仓库的开发者，这里有一份快速判断清单：

- ✅ 你用 AI 编程助手（Cursor / CodeBuddy / Claude Code / Windsurf）写代码
- ✅ 你的项目超过 10 个文件，经常改一处炸另一处
- ✅ 你遇到过同一个 Bug 反复出现
- ✅ 你的项目文档已经过时了
- ✅ 你希望有一套自动化的质量闸门

→ 这个工具适合你，30 秒安装，改 3 次代码就能感受到区别。

- ❌ 你只写 1-2 个文件的脚本
- ❌ 你不需要 AI 辅助编程
- ❌ 你在找 PDF/PPT/Excel 处理 Skill

→ 这个工具不适合你（但你可以去看看 claude-skills 的 354 个领域技能）。

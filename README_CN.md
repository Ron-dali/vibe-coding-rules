## 🎯 它到底解决什么问题？

用 AI 写代码的人，一定会遇到这 5 个痛点。不是一个两个，是**全部**：

| # | 痛点 | 具体表现 |
|:--|:--|:--|
| 1 | 🧠 **AI 失忆** | 昨天修好的 Bug，今天原样重现。改 B 模块时悄悄破坏了 A 模块，你不知道，直到用户骂。 |
| 2 | 🔁 **重复踩坑** | 同样的错误——`\|\|` 把 `0` 当成 falsy 吞掉、`<select>` 假选中没占位选项——犯了 5 次还在犯。AI 没有"吃过亏就记住"的能力。 |
| 3 | 📝 **改完不知道改了什么** | 没有变更记录。动了哪些文件？影响了什么？下次出问题连回溯的起点都没有。 |
| 4 | 🧪 **改完不知道坏了没有** | 改一行 CSS，整个页面布局崩了——手动回归测试太累，不测又不放心。 |
| 5 | 🗺️ **项目越大越迷路** | 打开一个文件，不知道它关联了谁、上次怎么改的、哪里踩过坑。每改一次都是一次探险。 |

**Vibe Coding Rules 把这 5 个痛点全部变成了自动化流程。你不需要懂编程，不需要记住任何规则——AI 自己会记住、会检查、会测试。**

## ⚡ 它是怎么运作的？——一个真实场景

假设你说：**"修复用户设置页面的保存按钮不起作用"**

以下是 AI 在装上 Vibe Coding Rules 之后的完整工作流程：

### 第一步：🧠 理解问题 + 面包屑预扫描（coding-principles 自动触发）

AI 不是直接开始改代码。它先停下来问自己：

- 保存按钮涉及哪些文件？
- 打开 `settings.js` 头部看到 🍞 面包屑写着：
  ```
  @COUPLED: server/routes/settings.js, public/js/validation.js
  @BUGFIX 2026-06-20: 表单字段名不匹配导致保存静默失败
  ```
- AI 立刻知道：不能只改前端，还要检查后端路由和验证逻辑；上次修过字段名不匹配的问题，别回退了。

**一行代码都没写，AI 已经知道了完整的影响范围。**

### 第二步：⚡ 改代码（安全终端保护）

如果需要调 API 验证，`safe-terminal-executor` 自动把 curl/wget 命令包裹成带超时保护的 Node 脚本——不会因为网络卡住而无限等待。

### 第三步：🔍 28 条规则自检（self-check 自动触发）

代码改完了，但 AI 不会直接说"好了"。它逐项检查 28 条编程纪律规则：

- ✅ `||` 没有吞掉数值 0？（陷阱 A）
- ✅ `<select>` 有没有占位 option？（陷阱 B）
- ✅ POST upsert 有没有硬编码字段？（陷阱 C）
- ✅ 双入口 HTML 的 `<script>` 标签两边都同步了？
- ✅ 🍞 面包屑更新了？
- ... 共 28 条，自动跑完。

**代码不是"看着对就行"，而是 28 条规则逐条过。**

### 第四步：🧪 自动化测试（web-testing 自动触发）

代码改了、检查过了——但 **AI 自己打开浏览器验证**：

- DOM 断言：保存按钮在不在？能不能点？
- 截图对比：设置页面布局有没有崩？
- OCR 文字验证：页面上写的文字对不对？

**改 CSS 不会悄无声息炸掉布局。AI 帮你亲眼看了。**

### 第五步：📝 变更日志（changelog 自动写入）

一切通过后，自动记录：

- 改了哪些文件
- 新增/删除多少行
- 28 条规则通过率是多少
- 有没有新规则被触发（从观察池晋升到硬规则）

### 第六步：✅ 闭环交付

```
coding-principles（改前自查）
  → 改代码（安全执行）
    → self-check（28条规则扫描）
      → web-testing（浏览器自动验证）
        → changelog（变更记录）
          → ✅ 交付
```

**你只需要说一句话。剩下的，AI 自己闭环。**

## 🍞 面包屑系统——整套工具的灵魂

这是 Vibe Coding Rules **最核心的机制**。每次改代码时，AI 在文件头部自动播种"面包屑"：

```javascript
/*
 * ─── 🍞 AI面包屑导航 ───────────────────────────────
 * 标签含义: @COUPLED=联动文件 @GOTCHA=陷阱 @BUGFIX=Bug修复 @MAGIC=魔数
 *          @DEPENDS=外部依赖 @ASSUME=假设 @TODO=待办 @WHY=选型理由
 *          @PERF=性能 @CONTRACT=接口约定 📖=开发文档引用
 *
 * 面包屑（改这里会影响谁）:
 *   @COUPLED 改字段需同步: public/js/chat-core.js#L200, desktop/renderer/chat.html
 *   📖 见: 开发文档/03-架构设计/工具注册流程.md
 *   @BUGFIX 2026-07-06: 修复截图base64被5000字符截断，恢复: history.js移除截断
 *   @GOTCHA: openTab() 里的状态机重置逻辑，改顺序会导致双击初始化
 * ────────────────────────────────────────────────
 */
```

**改前扫描**：打开文件先读 🍞 → 知道耦合关系和历史坑位 → 不会踩旧坑

**改后播种**：改完自动更新 🍞 → 下一个 AI（或未来的自己）打开文件时，有完整地图

> 面包屑为什么叫"灵魂"？因为 AI 的失忆不是靠"提醒"解决的——是靠**代码自己携带记忆**。每改一次文件就留一条🍞，改 50 次就形成了一张完整的项目地图。后面来的 AI 打开任何文件，都不需要你解释上下文了。

## 🎚️ 渐进式采纳（V2.7 新特性）

> **不是每次改代码都要跑 6 个 Skill。改 3 行 CSS 和改一个新功能，用的流程不一样。**

| 级别 | 触发条件 | 激活的 Skill | 适用场景 |
|:--|:--|:--|:--|
| **L1 · 轻量** | ≤ 3 个文件，≤ 30 行变更 | 五原则自检 | 修小 Bug、改文案、调 CSS |
| **L2 · 标准** | 3-10 个文件，或涉及路由/数据流 | L1 + 28 条规则自检 | 跨模块 Bug、中等改动 |
| **L3 · 完整** | > 10 个文件，或新功能/重构 | L1 + L2 + 自动测试 + 变更日志 | 新功能、架构调整 |

**不浪费上下文。** 改 3 行 CSS 不需要跑完整测试链路。

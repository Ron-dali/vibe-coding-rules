## 一、触发条件

以下任一情况发生时，**必须执行自检**：
- 修改工具目录下的文件
- 修改 schema（name、description、parameters）
- 修改兜底工具列表（如配置了三层注册）
- 修改核心链路代码
- 修改双入口 HTML（如配置）
- 修改 Express 路由
- 修改数据库 schema
- 修改 VERSION 文件
- 修改项目概览文档（README.md / OVERVIEW.md）

---

## 二、架构注册一致性检查（可选）

如 `pipeline.json` 中 `architecture.hasThreeLayerRegistration = true`，需检查工具目录与兜底列表一致性。

### 检查步骤

1. 列出工具目录所有文件
2. 提取兜底列表中所有 name 字段
3. 交叉比对：工具有而兜底无 → 补；兜底有而工具无 → 删
4. 确认数量一致

---

## 三、Schema 字段完整性

每个工具条目必须包含：
- `name` — 与源文件一致
- `description` — 一致（可精简，不遗漏关键信息）
- `parameters.type` — 固定 `"object"`
- `parameters.properties` — 与源文件一致
- `parameters.required` — 与源文件一致

---

## 四、数据传输链路截断检查

如 `pipeline.json` 中配置了 `architecture.truncationPoints`：

### 检查步骤

1. 搜索所有 `substring`、`slice`、`limit`、`truncate` 调用
2. 确认截断上限能容纳最大体积数据（如 base64 图片 1-5MB）
3. 用大体积数据端到端测试

---

## 五、全量 diff 铁律

1. **先读代码再下结论** — 不靠记忆
2. **全量比对不偷懒** — 列出全部、逐项交叉
3. **差集双向检查** — "A有B无" 和 "B有A无"
4. **改完复读确认** — 改后 read_file 验证
5. **注册 ≠ 可用** — 注册一致只是第一步，传输链路才是瓶颈

## 七、29条代码纪律规则

### 规则 1 — 先读代码再下结论
不靠记忆，必须 read_file/list_dir 确认现状。

### 规则 2 — 全量比对不偷懒
列出全部文件，逐项交叉比对。

### 规则 3 — 差集双向检查
"A层有B层无" 和 "B层有A层无" 都要检查。

### 规则 4 — 改完复读确认
replace_in_file 后 read_file 确认修改成功。

### 规则 5 — 截断点全量扫描
substring/slice/limit 上限能否容纳最大数据。

### 规则 6 — 用最大数据测试
小数据通过 ≠ 链路正常，必须用大数据端到端验证。

### 规则 7 — 注册 ≠ 可用
工具注册一致只是第一步，数据传输链路才是瓶颈。

### 规则 8 — 数值不用 `||` 判空
`value || default` 会吞 0/''/false。数值用 `!= null` 或 `??`。

### 规则 9 — `<select>` 必须加占位 option
缺省时加 `<option value="" disabled selected>请选择...</option>`。

### 规则 10 — POST 即 UPSERT 更新分支不可硬编码
从 req.body 取值，不可写死常量。

### 规则 11 — 自检规则自生长
修复新型 bug 后：抽象为规则追加 → 补录历史事件。

### 规则 12 — 浮层/抽屉/弹窗穷举关闭路径
检查 X按钮/ESC/遮罩点击/所有导航入口/DOM 清理。

### 规则 13 — Express 固定路由在参数化路由之前
`/api/xxx/specific` 必须在 `/api/xxx/:param` 之前。

### 规则 14 — 新增工具必须补前端显示名映射
格式化函数的 map 必须补条目。

### 规则 15 — 双入口脚本/DOM 同步
两个入口的 script 标签和 DOM 元素完全一致。

### 规则 16 — 组件布局迁移时旧工件清零
遮罩/position/z-index/挂载点/事件绑定全部清理。

### 规则 17 — 同类组件功能对等
多输入区间功能一致（类型/工具栏/功能能力）。

### 规则 18 — 可重复初始化容器子组件状态同步
innerHTML 重建前重置 `_rendered`/`_initialized` 等标记。

### 规则 19 — 状态快照恢复分离实体无关/相关
对话历史可恢复，系统消息/active 标记不可恢复。

### 规则 20 — 异步回调验证上下文一致
fetch/await/setTimeout 中检查实体 ID 是否匹配。

### 规则 21 — 自检通过后必须更新变更日志
不可跳过或"回头再补"。

### 规则 22 — 双入口 DOM 元素同步
一处新增的 DOM 必须同步到另一处。

### 规则 23 — npm 依赖必须检查 ESM/CJS 兼容性
ESM-only 包在 CJS 项目 require() 会报错。

### 规则 24 — 同类 API 端点必须使用相同数据源
GET(统计)和 POST(操作)的路径解析逻辑必须一致。

### 规则 25 — [HARD] 项目概览文档版本一致性（V2.2 新增）

**检查项目概览文档（README.md / OVERVIEW.md）中的版本号是否过时：**

1. 读取 `.ai-pipeline/VERSION`（或项目根目录 `VERSION`）获取真实版本号
2. 搜索项目根目录概览文档（优先 `OVERVIEW.md`，其次 `README.md`）中的版本号模式（`v\d+\.\d+`、`版本 \d+` 等）
3. 如果概览文档中存在版本号且与 VERSION 不一致 → **[HARD] 阻断**，必须同步更新
4. 如果概览文档中写的是"当前版本见 `.ai-pipeline/VERSION`" → 通过（这是最佳实践）
5. 如果项目概览文档不存在 → 通过

**目的**：对抗 AI 生成文档的自然腐烂——AI 在一次会话中创建的项目概览文档（如 OVERVIEW.md），在后续会话中永远不会被更新，版本号逐渐过时，最终变成误导信息。

### 规则 26 — 设计意图锚点注释（V2.3 新增）

**代码即说明书——每次改动必须在关键位置留下设计意图注释。**

10 种标准化标签：

| 标签 | 用途 | 示例 |
|:--|:--|:--|
| `@WHY` | 方案选型理由 | `@WHY 用EventEmitter而非Redux：只有3个订阅者` |
| `@GOTCHA` | 反直觉但故意的写法 | `@GOTCHA 别优化成并行：服务端限流5并发` |
| `@MAGIC` | 魔法数字含义 | `@MAGIC 30=支付超时上限(秒)` |
| `@COUPLED` | 隐式耦合警告 | `@COUPLED 改这里需同步改: fileA, fileB` |
| `@BUGFIX` | Bug修复记录 | `@BUGFIX 2026-03-15 重复扣款，修复:乐观锁` |
| `@PERF` | 性能敏感区域 | `@PERF O(n²)但n<10，瓶颈不在这` |
| `@ASSUME` | 数据假设 | `@ASSUME userId是数字字符串` |
| `@TODO` | 带日期的临时方案 | `@TODO 2026-09-01 切新版登录方案` |
| `@EXPIRE` | 过期倒计时 | `@EXPIRE 2026-09-01 旧API即将下线` |
| `📖` | 文档联动入口 | `📖 完整设计: docs/design/payment.md` |

**检查步骤**：
- 阶段A（改前）：扫描目标文件已有锚点 → 预判需要写什么标签
- 阶段B（改后）：验证新增/修改的代码是否包含必要的设计意图注释

详细规范见 `ANCHOR_SPEC.md`。

### 规则 27 — 注释-文档联动一致性（V2.3 新增）

**代码注释中的 📖 引用和文档中的「实现位置」必须双向一致，形成不可打破的回环。**

四个方向验证：

**方向1 — 代码→文档**：代码中的 `📖` 路径 → 文档必须存在且内容匹配
**方向2 — 文档→代码**：文档「实现位置」→ 代码路径必须存在
**方向3 — 文档→管理文档**：文档表头的 `📋` 引用 → 管理文档必须存在
**方向4 — 闭环验证**：代码→📖→文档→📋→管理文档→索引→代码，四环闭合

任一断裂 → 修复（阻断 if 路径不存在，警告 if 内容不匹配）。

### 规则 28 — 🍞 面包屑自动播种（V2.5 新增）

**每次代码改动必须给每一个被修改的文件补设或更新 🍞 AI面包屑导航头。**

**阶段A — 文件无 🍞 头部 → 创建**：
在文件开头插入标准格式导航块，自动填入：
- a) `@COUPLED`：本次会话中同时修改了哪些其他文件？→ 相互添加指向
- b) `📖`：本次改动过程中读取了哪些开发文档？→ 指向它们
- c) `@BUGFIX`：本次是 Bug 修复吗？→ 添加日期 + 原因 + 方案
- d) `@GOTCHA`：有反直觉/易踩坑的写法吗？→ 标注陷阱
- e) `@MAGIC`：有硬编码的业务数字吗？→ 标注含义

**阶段B — 文件已有 🍞 头部 → 更新**：
- f) 验证已有 `@COUPLED` 指向的文件路径是否仍存在
- g) 验证已有 `📖` 指向的文档路径是否仍存在
- h) 新增本次改动产生的新面包屑
- i) 删除过期/失效的面包屑，或标记 `@EXPIRE`

**🍞 头部标准格式**（内嵌标签含义表，任何 AI 无需外部文档即可理解）：

```
/*
 * ─── 🍞 AI Breadcrumb Navigation ──────────────────
 * Tag meanings: @COUPLED=linked files @GOTCHA=gotcha @BUGFIX=bug fix @MAGIC=magic number
 *              @DEPENDS=external dependency @ASSUME=assumption @TODO=todo @WHY=design rationale
 *              @PERF=performance @CONTRACT=interface contract 📖=dev doc reference
 * 
 * Breadcrumbs (changing this affects):
 *   @COUPLED sync needed: fileA, fileB
 *   📖 see: docs/xxx/yyy.md
 * ──────────────────────────────────────────────────
 */
```

**面包屑质量要求**（不符则阻断）：
- `@COUPLED` 必须具体到文件路径，不可笼统写"所有前端文件"
- `📖` 指向的文档必须真实存在
- `@BUGFIX` 必须包含日期 + 简短原因 + 修复方案

### 规则 29 — 📏 文件行数检查（V2.9 新增）

**每次代码改动后，检查被修改文件的功能代码行数是否超过 300 行红线。**

**计数规则**：
- a) 🍞 面包屑头部（约 15-20 行）不计入
- b) `use strict` / shebang / 许可证声明不计入
- c) 纯注释行（无 `@TAG` 的业务性注释）不计入
- d) 空行不计入
- e) 其余所有代码行（函数、变量、class、import/require 等）计入

**检查规则**：
- f) 改动后 > 300 行 → ⚠️ 警告，建议拆分（阻断级别：[SOFT]）
- g) 改动后 > 500 行 → 🔴 强制提醒（阻断级别：[HARD] for 非豁免类型）
- h) 检查是否在豁免列表中（配置/声明 ≤ 500、高度内聚算法 ≤ 400、框架生成不计入）
- i) 新增功能导致原文件 > 300 行 → 应新建文件/模块而非追加

**与前置门的配合**：前置门门3 是改前判断，本规则是改后自动化检查——不再依赖 AI 记忆。

**插入位置规则**：
- JavaScript/TypeScript: `'use strict';` 之后，第一个 `import/require` 之前
- HTML: `<!DOCTYPE html>` 之后，`<head>` 之前（用 `<!-- -->` 注释）
- CSS: 文件第一行（用 `/* */` 注释）
- Python: `#!` shebang 之后，第一个 `import` 之前（用 `#` 逐行，🍞 替换为 `[breadcrumb]`）
- Markdown: `#` 标题之后，正文之前（用 `<!-- -->` HTML注释）

此规则的独特价值：面包屑从"需要人工刻意耕种"变成"每次改动自动播种"，内嵌的标签含义表保证了即使 AI 没加载本 Skill，看到 🍞 也能理解每个标签的用途——自解释文档的核心设计原则。
